# HU-13 — Login y autorizacion · Checklist manual

Casos de prueba del Plan de Pruebas v2.0 que aplican al HMI (cliente React + bridge Node).
Los casos que dependen del backend Java (firma JWT, validacion de rol, expiracion server-side)
estan fuera del alcance de este checklist y se cubren en el sprint integrado del backend.

## CP-LOGIN-01 — Login exitoso con usuario supervisor

**Tipo:** Funcional + Seguridad
**Precondicion:** Backend Spring Boot arriba. BD con `alopez/5678/supervisor`.

### Pasos
1. Abrir `http://localhost:5173/` en navegador limpio (sin sesion).
2. Verificar redireccion automatica a `/login` — el panel **no** debe ser visible.
3. Ingresar `usuario=alopez` y `password=5678`. Click en **Entrar**.
4. Verificar que la URL cambia a `/` y se renderiza el panel con los 5 cuartos.
5. Abrir DevTools → Application → Session Storage → confirmar `sei.auth.token` presente.
6. Pegar el token en `https://jwt.io` y verificar claims: `operador_id`, `rol='supervisor'`, `exp` futuro.

**Resultado esperado:** login exitoso, JWT almacenado, panel renderizado, claims validos.

## CP-LOGIN-02 — Login fallido con credenciales invalidas

**Tipo:** Funcional + Seguridad
**Precondicion:** Backend arriba.

### Pasos
1. Estando en `/login`, ingresar `usuario=alopez` y `password=incorrecto`. Click en **Entrar**.
2. Verificar mensaje de error visible: **"Usuario o contrasena incorrectos."**
3. DevTools → Network → confirmar que `POST /api/login` retorna `401` o `403`.
4. Session Storage NO debe tener `sei.auth.token`.
5. Repetir con `usuario=no_existe` / `password=cualquiera`. El mensaje debe ser identico (no filtrar si el usuario existe).

**Resultado esperado:** rechazo claro, sin sesion creada, mensaje uniforme para ambos escenarios.

## CP-AUTH-01 — Acceso a ruta protegida sin sesion (HMI)

**Tipo:** Funcional + Seguridad
**Precondicion:** Sin sesion activa (`sessionStorage` vacio).

### Pasos
1. Limpiar `sessionStorage` desde DevTools y recargar.
2. Intentar navegar manualmente a `/` o `/cualquier-ruta`.
3. Verificar redireccion inmediata a `/login`.
4. DevTools → React DevTools → confirmar que `useAuth().isAuthenticated === false`.

**Resultado esperado:** ninguna ruta protegida es accesible sin sesion. El catch-all redirige al login.

## CP-AUTH-03 — JWT expirado fuerza logout en HMI

**Tipo:** Funcional + Seguridad
**Precondicion:** Backend con `jwt.expiracion=60s` (modo prueba). Usuario logueado.

### Pasos
1. Hacer login como `alopez`.
2. Verificar que `useAuth().user.expiresAt - Date.now()` esta en orden de 60s.
3. Esperar 65 segundos sin actividad.
4. Verificar que el HMI redirige automaticamente a `/login` (timer interno de `AuthContext`).
5. Confirmar que `sessionStorage.sei.auth.token` fue limpiado.
6. Re-login exitoso debe restaurar el panel.

**Resultado esperado:** el `setTimeout` programado al login dispara `logout` exactamente al vencer `exp`. No quedan tokens vencidos en storage.

## Verificaciones complementarias del HMI

### Persistencia de sesion al recargar
1. Estando logueado, presionar F5.
2. La pagina recarga sin pasar por `/login`. El usuario sigue identificado.

**Estado:** [ ] Pass  [ ] Fail

### Token con forma invalida
1. En DevTools, ejecutar `sessionStorage.setItem('sei.auth.token', 'no-es-jwt')` y recargar.
2. La app debe redirigir a `/login` y limpiar el storage.

**Estado:** [ ] Pass  [ ] Fail

### Rechazo del bridge sin JWT
1. Sin sesion, abrir consola y emitir manualmente `socket.emit('silenciar_alarma', {cuartoId: 1})` contra el bridge.
2. En logs del server: `silenciar_alarma rechazado: jwt_token ausente o invalido`.
3. EMQX Dashboard NO muestra publicacion en `sei/cuartos/1/alarma/cmd`.

**Estado:** [ ] Pass  [ ] Fail

### Header Authorization en llamadas REST
1. Logueado, abrir DevTools → Network y disparar cualquier request via `httpClient`.
2. Header `Authorization: Bearer <token>` presente.

**Estado:** [ ] Pass  [ ] Fail
